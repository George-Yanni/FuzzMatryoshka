# Test Plan

Command tip: run each command on its own line. If you prefer one line, use `&&` between commands.

## 1. Firmware Image Parser (`src/fw_parser.cpp`)

**Target:**

```bash
./matryoshka parse < image.bin
```

- **Valid Header:** Test with a perfectly valid image (generated by `make seed`).
- **Magic Number:** Change the first 4 bytes. Parser should reject it.
- **Version Mismatch:** Change the format version field to something unsupported (e.g., 2 or 0).
- **Truncated File:** Provide a file smaller than the header size (less than 36 bytes).
- **Excessive Segments:** Set `num_segments` to a value larger than `FW_MAX_SEGMENTS` (32). This is a likely overflow candidate.
- **Buffer Overread:** Set `header_len` or `payload_len` to values larger than the actual file size.
- **Digest Verification:** Modify a single byte in the payload without updating the digest. Verification should fail.
- **Rollback Protection:** Try to apply an image with `fw_version` lower than the device's current version.

## 2. Update Protocol & State Machine (`src/protocol.cpp`)

**Target:**

```bash
./matryoshka protocol
```

- **Happy Path:** Send `START <size>` → `CHUNK ...` → `COMMIT`. Device should update.
- **State Violation:** Send `CHUNK` or `COMMIT` before `START`. Should be ignored or handled gracefully.
- **Double Start:** Send `START` while already in `Receiving` state. Should reset/restart or error.
- **Abort:** Send `START` → `CHUNK` → `ABORT`. State should return to `Idle` and buffer should clear.
- **Invalid Hex:** Send `CHUNK` with non-hex characters (e.g., `CHUNK 0 ZZ`).
- **Out of Bounds Chunk:** Send a `CHUNK` with an offset larger than the `START` size.
- **Negative/Huge Numbers:** Send `START -1` or `START 999999999999` to test integer parsing.
- **Memory Exhaustion:** Send a `START` with a very large size (e.g., 2GB) to see if allocation fails safely.

## 3. Updater Tool (`src/updater.cpp`)

**Target:**

```bash
./updater < image.bin
```

- **Output Format:** Verify it prints valid `START`, `CHUNK`, and `COMMIT` lines.
- **Chunking:** Ensure it splits large files into multiple `CHUNK` lines correctly.
- **File Read Error:** Run against a non-existent file.
- **Empty File:** Run against a 0-byte file.

## 4. Integration (End-to-End)

**Target:**

```bash
./updater image.bin | ./matryoshka protocol
```

- **Full Update:** Pipe the updater output directly to the device. The device should report the new version upon completion.
- **Partial Stream:** Kill the updater halfway through. The device should remain in `Receiving` state or time out (if timeout existed) but not crash.

## 5. Fuzzing Vectors (for radamsa)

Since your goal is to fuzz, these are the specific inputs you will generate:

### Binary Fuzzing

- **Input:** `seeds/valid_fw.bin`
- **Target:** `./matryoshka parse`
- **Goal:** Find crashes in `parse_firmware_image`.

### Protocol Fuzzing

- **Input:** A text file containing valid protocol commands.
- **Target:** `./matryoshka protocol`
- **Goal:** Find crashes in `process_line`, `cmd_chunk`, or `parse_hex_byte`.

### Chunk Data Fuzzing

- **Input:** A valid `CHUNK` line.
- **Target:** `./matryoshka chunk`
- **Goal:** Test the hex parsing logic specifically.
